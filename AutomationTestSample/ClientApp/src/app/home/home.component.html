Automation Test Sample Application

This is a sample application to be used for automation tests.

It uses an in-memory database for managing data for this application. As soon as the application shuts down, all data is lost.

Overview

Everlight is developing a worklist application for managing orders coming in from clients. The MVP involves Everlight's operations team taking orders over the phone, entering them into this system, and then managing the orders from the worklist. The application is partially developed at this stage, but all specifications below refer to the features that are currently implemented.

Instructions

  1. Manual Test: List down the test scenarios, test the application and create a bug report
  2. Automation: Create a framework that supports UI and API-based automation that can run from the configured browser (Chrome/Firefox/Edge), with implicit and explicit logging and reporting

Below is the data model and features for the application in its current version, plus definitions for domain language. You should implement a testing framework that validates the expected behaviour.

Click on the Orders menu on top right to get started.




Data Model

  • A client is a hospital or radiology practice. They are often refered to by their organisation code.
  • An order includes a study (a set of images) scanned at the client's site, such as an xray or MRI, for which Everlight will produce a clinical report
  • The Patient's Mrn is the hospital's Medical Record Number, essentially the patient's unique identifier. Interestingly is it not required to be an actual number by any healthcare protocols.
  • The Order's AccessionNumber is an identifier from the client's systems, grouping one or more studies together. For this exercise, we are assuming there is only ever 1 study in an accession and order.
  • The modality is the type of study, such as an xray or MRI.
  • There are several statuses in a teleradiology workflow:
    • SC - the order is scheduled
    • IP - images are in progress of being sent
    • CM - images are complete
    • NG - the study is not for reporting
    • CMD - the report is in transcription
    • ZZ - the report is ready
    • CA - the order has been cancedlled

Features

  • The Orders page shows a list of pre-created orders. These are ordered by the study datetime â€“ the patient that was scanned earliest is the highest priority.
  • There is an option to cancel an order, which currently just deletes the order. This, however, should not delete the patient that it is attached to, in case that patient is attached to other orders.
  • A button called Create New can be used to go to Order Form page which is used to create a new order.
  • When creating a new order, the status is automatically set to SC. The development team is yet to implement any features related to changing the status.
  • If the MRN matches an existing patient, then that patient record will get linked to the new order.
  • The API verbs for Patient and Order have been implemented, and can be reached at api/Orders and api/Patients respectively.
  • The API verbs for Client and Site are yet to be implemented. Status and Modality are fairly static within the domain and so API verbs are not likely to be implemented in the applicatin; any changes required would be handled by the development team.
